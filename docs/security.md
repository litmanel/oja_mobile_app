# security.md

Security posture for Oja Market v1. Threats are specific to Nigeria tier-2 context: SIM swap fraud, false-report abuse, coordinated vendor suppression.

---

## Authentication

**Method:** Phone number + OTP (Supabase Auth)  
**JWT:** Issued on OTP verification. Stored in secure storage on device (`flutter_secure_storage`), never in SharedPreferences.  
**Session:** JWT expiry 7 days. Refresh token rotation enabled.  
**New-device login:** Triggers WhatsApp alert to the phone number. Vendor is informed but cannot block the login in v1 (full device management is v2).

### SIM swap — known partial gap in v1

SIM swap is a real threat in the Nigerian context. v1 mitigation is limited to:
- New-device login alert via WhatsApp (attacker receives this too if they control the SIM)
- Account recovery requires human verification via market association contact or in-person — no automated SIM-only recovery

**Full mitigation (e.g. secondary email, device binding, hardware key) is a v2 security feature.** This risk is accepted and documented. Do not close it without a formal threat model review.

## Data validation

All input validated server-side in Edge Functions before any DB write. Client-side validation is UX only — never a security boundary.

| Field | Rule |
|-------|------|
| `listings.name` | String, 3–80 chars, strip HTML |
| `listings.price` | Positive decimal, max 9,999,999 |
| `listings.unit_label` | Enum value or custom string ≤20 chars |
| `listings.quantity` | Positive integer |
| `listings.image_url` | Cloudinary `public_id` format only |
| `users.phone` | E.164 format, Nigerian prefixes only (`+234`) |
| `shops.market_name` | Enum or `market_name_custom` ≤60 chars |
| `reports.reason` | Enum — no free-text in v1 |

No field accepts raw HTML. Strip on ingest.

---

## Image uploads

- Uploads go directly from device to Cloudinary via **signed URL** — never through the application server
- Signed URLs generated by Edge Function, valid for 60 seconds, single-use
- Unsigned uploads rejected at the Cloudinary level (upload preset set to `signed` mode)
- MIME type validated server-side: JPEG, PNG, WebP only
- Max file size: 5 MB

## Row Level Security (RLS)

All Supabase tables have RLS enabled. No table is publicly writable without an authenticated JWT.

Key policies:

| Table | Read | Write |
|-------|------|-------|
| `listings` | Public (active, non-deleted) | Owner only |
| `shops` | Public | Owner only |
| `users` | Own record only | Own record only |
| `contact_taps` | Admin only | Authenticated (insert own) |
| `reports` | Admin only | Authenticated (insert own) |

Admin role (`users.role = 'admin'`) bypasses read restrictions. Does not bypass audit logging.

## Moderation & false reporting

**Coordinated suppression:** If ≥2 of 3 reports on a listing come from the same `reporter_id`, the cluster is flagged for human review rather than triggering auto-hide.  
**Vendor appeal:** Suspended vendors can submit one appeal. Appeal reviewed by admin within 48 h (target). Outcome delivered via WhatsApp.  
**Audit trail:** All moderation actions (hide, suspend, reinstate) write an immutable log entry with admin `user_id`, timestamp, and reason.

## NDPA compliance (Nigeria Data Protection Act)

**Threshold:** At 500 vendors + estimated buyer accounts, data subjects exceed 2,000 — DPO designation is **required at launch**.

| Obligation | Status |
|-----------|--------|
| DPO designated | Required before launch |
| Privacy policy published | Required before launch |
| Data retention periods defined | See below |
| Breach notification (72 h) | Operational procedure required |
| Legal review | Budget ~₦150,000–300,000 estimated |

**Retention periods:**

| Data type | Retention |
|-----------|-----------|
| User PII (name, phone) | Until deletion request, then anonymised immediately |
| Listing content | 90 days after expiry, then purged |
| `contact_taps` | 12 months (analytics) |
| Transaction-adjacent records | Per CBN mandate (not applicable v1, but noted for v2) |
| `worker_errors` logs | 30 days |

**Deletion flow:** Anonymises PII fields — does not hard-delete rows. Retains anonymised aggregates. See `AGENTS.md` → Data Deletion Worker.

## Transport

- All API communication over HTTPS — no HTTP fallback
- Certificate pinning: evaluate for v2. Not implemented in v1 (adds complexity to OTA update flow)
- Supabase Edge Functions enforce HTTPS at the platform level

---

## Secrets management

- No secrets in source code or Flutter assets
- Supabase URL and anon key in environment variables, injected at build time via CI
- Cloudinary upload preset name is not secret — signed URL generation key is (stored as Supabase Edge Function environment variable)
- Rotate all secrets if a breach is suspected — documented in incident response runbook (maintain separately)

| Risk | Accepted? | Path |
|------|-----------|------|
| SIM swap | Accepted with alert | v2 |
| Device binding / 2FA | Not implemented | v2 |
| Certificate pinning | Not implemented | v2 |
